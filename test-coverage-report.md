## Known flaky functions

### `runScenario()` in `studio/app.js` — highest risk

Browser-only; zero automated test coverage. Flakiness sources:

- **Network dependency**: calls `POST /inject` and `GET /state` against a live
  runtime. Fails silently if the runtime is not running (shows error toast, no
  assertion).
- **State accumulation**: injects messages into the *running* runtime without
  resetting between steps, so a failed step leaves the runtime in a dirty state
  that affects all subsequent steps in the same run.
- **No reset between runs**: clicking Run twice accumulates state. A passing
  first run may become a failing second run.
- **Operator coercion edge cases** in `evalOp`: `equals` uses `==` after
  coercing numeric-looking strings with `Number()`. Mixed comparisons like
  `actual=9.99 expected='9.99'` work; `actual=null expected='null'` will not
  (`null == 'null'` is `false`).

### `getFieldValue()` in `studio/app.js` — silent wrong answers

Resolves dot-paths like `cart.totals.subtotal` into the live state object.
When it encounters an array it takes `[0]` (correct for single-row record
tables). For multi-row indexed tables (e.g. `cart.items`) it silently returns
the first row only, giving wrong comparison results without any error.

### HTTP integration tests — port conflict flakiness

The four `describe` blocks that spawn `flux run` all use hardcoded ports
(`14_099`, `14_100`, `14_101`). If a prior test run was killed mid-flight and
left a server still listening, the new `before()` spawn fails with EADDRINUSE
and the entire suite's tests fail. Observed in practice: running the test suite
twice in quick succession can reproduce this.

Mitigation in place: each `after()` calls `server?.kill()`. Residual issue:
`kill()` on a process that has already exited is a no-op (no error), but a
process killed with SIGKILL during cleanup may not release the port immediately.

### `POST /scenario/run` — test-ordering sensitivity (fixed)

`test_autogenerated.scenario.yaml` written by the round-trip test persisted
into subsequent test runs. Fixed: `before()` and `after()` hooks in the Studio
API describe block now delete any `test_autogenerated*.scenario.yaml` files
from the fixture's `scenarios/` directory.

### `saveUnitSource()` / `PUT /unit/:name/source` — writes to source on disk

No integration test for the write path (would mutate fixture files). The route
exists and handles the request, but any write failure is only surfaced via a
toast notification. Confirmed working manually but not in CI.

```js
if (/\bbool\b/.test(str)) return false   // ← not tested
return null                               // ← bare annotation with no type keyword
```

`bool`-typed state fields and a bare-`null` annotation are valid but rare.
Low risk — all common types (`u32`, `f32`, `string`, `null`) are covered.

### `src/scenario-runner.js` lines 109–110 — `checkMessageAssertion` fallthrough

Reached when `expect_message` specifies a field assertion and a message with
the right topic was emitted but the field value did not match. The simpler
paths (no message emitted; message with correct field) are covered.

### `src/scenario-runner.js` lines 193–194 — `formatResults` failure indent

Reached when `formatResults` renders a scenario that has failing steps with
individual `failures[]` strings. The aggregate pass/fail display is covered;
this detail indentation path is not.

### `src/checkpoint.js` branch gap (87.10%)
